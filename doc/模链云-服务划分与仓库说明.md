## 1. 文档目的

固定 **仓库个数**、**后端运行时划分** 与 **入口路径**，避免与「微服务过度拆分」或「单体混杂路由」混淆。**后续对话与实现请勿偏离本节**，变更须更新本文版本号。

---

## 2. 仓库层面：四个交付单元（名称已定）

| 仓库 | 类型 | 职责 |
|------|------|------|
| **modlink-cloud** | 总仓 / 文档与编排 | 顶层文档（含 `doc/`）、跨仓库约定、版本矩阵、部署与 CI 说明；可含 git submodule 指向各子仓库 |
| **modlink-gateway** | 后端 | **逻辑上包含两个可部署进程**（见 §3）：**Gateway**（`/mlk/v1`）与 **Platform**（`/mlk/platform/v1`）。推荐在同一仓库内用 **`cmd/gateway` + `cmd/platform`**（或 `internal/` 分包）构建 **两个二进制**，单机运维最简单；**不强制**增加第五仓库。 |
| **modlink-cloud-admin** | 前端 — 管理后台 | Vue 3 + Vite + TS + Element Plus；**只调用** Platform API |
| **modlink-cloud-web** | 前端 — 用户门户 | 同上；**只调用** Platform API |

> **勿混淆**：对外仍是 **两个后端服务、两条路径**；只是在 **四个仓库** 约束下，Platform 服务端代码 **默认放在 `modlink-gateway` 仓库内** 与网关并列，避免无名增多仓库。若日后体量过大再拆独立后端仓库，属演进而非当前定稿。

---

## 3. 运行时层面：两个后端服务（逻辑边界）

| 服务名（逻辑） | URL 前缀 | 职责 |
|----------------|----------|------|
| **Gateway** | `https://api.{域名}/mlk/v1` | API Key 鉴权、转发 OpenRouter、SSE、与计费/风控联动 |
| **Platform** | `https://api.{域名}/mlk/platform/v1` | JWT、用户/租户/Key/钱包/订单/支付回调、管理端、报表、日志查询 |

单机 / 共机部署：**优先** `location ^~ /mlk/` 整体反代至本服务；服务内需 **进程或路由** 区分 **`/mlk/v1`** 与 **`/mlk/platform/v1`**（或合并单进程）。  
允许 **同一二进制内注册两套路由**（单进程），仍须在运维上视为 **两条逻辑服务**，便于日后拆机。

---

## 4. 前端与入口约定（勿忘）

| 客户端 | 调用对象 |
|--------|----------|
| **admin / web** | **仅** `…/mlk/platform/v1` |
| **终端用户业务系统（开发者集成）** | **仅** `…/mlk/v1`（Bearer `mk_live_` / `mk_test_`） |

前端 **从不** 直连 OpenRouter；上游密钥 **仅** Platform 配置存储，经 Gateway 注入上游请求。

---

## 5. 数据与中间件（摘要）

| 组件 | 主要使用者 | 说明 |
|------|------------|------|
| **MySQL** | Platform 为主；账务流水、订单、用户、租户 | Gateway 扣费需一致性与幂等时 **读库或调 Platform HTTP 内部接口**（实现择一，须在详细设计中写清） |
| **Redis** | Gateway + Platform | 会话、限流、幂等键等 |
