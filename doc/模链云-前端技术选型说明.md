## 1. 结论摘要

| 项目 | 选型 |
|------|------|
| **框架** | **Vue 3** |
| **构建** | **Vite** |
| **语言** | **TypeScript** |
| **UI 组件库** | **Element Plus**（管理端与用户端统一，降低学习与视觉割裂） |
| **工程形态** | **两个独立 Web 应用**：管理后台、用户前台各一个 Vite 工程（可分属两个 Git 仓库，与 PRD 一致） |
| **刻意不采用** | **uni-app** 作为 admin/web 主技术栈（理由见 §3） |

## 2. 与现有项目 `travel-together` 的关系

路径参考：`…/lv_ju/travel-together`（**uni-app + Vue 3 + Vite**，面向小程序多端 + H5）。

| 维度 | travel-together | 模链云 admin / web |
|------|-----------------|---------------------|
| **一致** | 同属 **Vue 3** 生态；团队可沿用组合式 API、Vite 思维、TS 习惯、代码规范工具链 | 降低认知成本 |
| **不一致** | **uni-app** 多端运行时 | **标准浏览器 SPA**：表格、大屏、文档站、充值流程以 **PC Web** 为主 |
| **实践建议** | — | 可复用 **HTTP 封装、类型约定、eslint/prettier 配置思路**；**不复制 uni 工程结构** |

---

## 3. 为何不采用 uni-app 承载 admin / web

1. **产品形态**：PRD 一期以 **Web 控制台 + 数据大屏 + 调用文档** 为主，非小程序优先。  
2. **组件与交互**：复杂表格、看板、宽屏布局、Markdown/代码示例等，**桌面端组件库**更合适。  
3. **维护成本**：admin 与 web 若强行塞进 uni-app，条件编译与多端差异会拖累迭代。

**若未来需要「模链云」微信小程序**：可 **单独** 新建 uni-app（或仅小程序端）工程，调用同一套后端 HTTP API；与 travel-together 的技术栈会更接近，但与本期 **modlink-cloud-admin / web** 分离。

## 4. 仓库与产物

| 仓库 | 说明 |
|------|------|
| **modlink-cloud-admin** | 运营/管理员：用户与租户、渠道、模型、订单、大屏、系统配置等 |
| **modlink-cloud-web** | 终端用户与开发者：账号、Key、充值、用量账单、调用文档 |

必要时可使用 **pnpm workspace / monorepo** 共享 `packages/ui` 或 `packages/api-client`，**非一期强制**。

---

## 5. 与其他架构决策的衔接

- **网关 Base URL**：用户文档中引导将 OpenAI SDK 指向 `https://api.<域名>/v1`（见 PRD §17.3）；前端仅调用 **业务 REST API**（登录、订单、用量等），**不直连 OpenRouter**。  
- **部署**：静态资源可由 Nginx/OSS + CDN 托管；与单机后端分离部署即可。
